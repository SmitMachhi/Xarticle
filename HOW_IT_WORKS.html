Section 01

What happens when you paste a URL

Every request takes one of two paths — a blazing fast cached path, or a fresh fetch path. Here's the full journey.

📋
You paste the URL
The moment you paste, your browser sends the URL to a tiny piece of code running on Cloudflare's network — literally on a server that's probably in your city. Not somewhere in Virginia. Your city.
🏷️
Is this URL in our memory? Cache Check
Before doing any real work, the backend checks: "Have we seen this exact URL in the last 5 minutes?" If yes — it returns the saved result instantly. Zero calls to X/Twitter. Zero parsing. Pure memory read. This path takes under 50ms.
🔑
Get the keys to X's API Cached 6 hrs
X's API is private and needs a special password (called a bearer token + query ID). These expire. Our backend caches them for 6 hours — so most requests skip the "find the password" step entirely. If they expire mid-request, it automatically finds fresh ones and retries. You never see this happen.
🏎️
Race two routes to X's servers simultaneously Hedged Fetch
Here's the trick that makes it reliable: the backend sends the request to two different X API addresses at the same time — like sending two taxi drivers to the same destination. Whoever arrives first wins. The slower one is cancelled. The 350ms stagger means we don't waste resources, but also never get stuck waiting on a dead endpoint.
🔬
Parse X's complex response into clean data
X's raw API response is like a 500-piece puzzle thrown in a bag. Our code picks out exactly what we need — title, author, engagement numbers, article blocks, images — and assembles them into a clean, structured format that the frontend can use immediately.
💾
Save to memory + send to you 5 min cache
The result is saved for 5 minutes so the next person who requests the same article gets it instantly. Then it's sent to your browser and you see the article appear.
Section 02

Four invisible speed layers

The backend doesn't just cache one thing. It has four separate memory layers — each one prevents a different type of slow work.

📦
Full Article Cache
Lives for 5 minutes
The whole finished result, ready to serve. Someone requested this article already? You get it in milliseconds. No API call needed at all.
🔐
API Password Cache
Lives for 6 hours
Getting X's API password requires scraping the X homepage and reading their JavaScript files. We do this once, save the result, and reuse it for 6 hours.
🎫
Guest Ticket Cache
Lives for 2 hours
X also requires a short-lived "guest ticket" for every session. We get one, save it, and reuse it for 2 hours instead of asking for a new one every request.
🖼️
Image Cache
Lives for 24 hours
Profile photos and article images are saved in your browser for 24 hours. Revisit the same article? Images appear before you even blink.
Section 03

The two-taxi trick

The single cleverest thing in the backend: racing two API routes against each other so a bad server never makes you wait.

🟢 Primary Route
api.twitter.com/graphql/…
Starts immediately
The original Twitter API
VS
🔵 Backup Route
api.x.com/graphql/…
Starts 350ms later
The newer X API address
Why 350ms delay on the backup? — If we fired both instantly every time, we'd double our API costs and potentially get rate-limited. The 350ms gap is the Goldilocks window: short enough that if the primary is dead or slow we barely notice, long enough that we don't waste the backup request when things work fine.

Section 05

The coffee shop analogy

Imagine Xarticle.co is a coffee shop

A bad coffee shop starts grinding beans from scratch every time someone orders a latte. Every customer waits 8 minutes. The barista has to walk to the storage room, find the beans, grind them, froth milk, do the whole thing fresh.

A smart coffee shop (ours) brews a fresh batch every 5 minutes and keeps it hot. Someone orders? Poured in 10 seconds. The barista already knows where everything is (cached tokens), has two espresso machines ready (dual API endpoints), and the menu never changes so she doesn't need to think (structured response parsing).

The Cloudflare part? That's like having a coffee shop on every block in the city instead of one downtown. You always walk to the one nearest you. That's why the first byte of response arrives so fast — it literally comes from a computer physically close to you, not from a data center across the country.

Section 07

What the backend is actually made of

Three components. No servers to manage. No databases. Just code and edge.

🌐
Cloudflare Pages
Hosts the website itself (what you see in your browser). Serves from the nearest location to you automatically.
Frontend
⚡
Cloudflare Worker
A tiny piece of JavaScript running at the edge — no traditional "server". Handles all API calls to X, caches results, proxies images. Starts in under 1ms (no boot time).
Backend
🧠
Cloudflare Cache API
Built-in memory storage at the edge. This is what stores all four cache layers — free, fast, and automatic. Not a separate service. It's just part of where the worker runs.
Memory
🐦
X / Twitter GraphQL API
The only external dependency. We call this to get article content. Everything else is our own code running at the edge.
External
Built for Xarticle.co — a tool for turning X article URLs into beautiful PDFs and Markdown.
